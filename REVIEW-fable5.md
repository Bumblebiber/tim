# TIM Codebase Review (Fable 5)

> Reviewed at HEAD `f0d4bb2` on branch `feature/plan-4-mcp-surface`, 2026-07-03.
> Method: three parallel deep-dive passes (store/core/search/migrate, mcp/cli, hooks/summarizer/sync), full `npm test` run, read-only `tim doctor` against `~/.tim/tim.db`, and cross-verification of every contested claim against HEAD source. Claims cite file:line; anything not confirmed by a tool result in this session is marked **unverified**.

## TL;DR

TIM's core bet — SQLite + FTS5, a hypergraph store, MCP tools on top, background summarization — is sound and the implementation is in much better shape than `docs/tim-capabilities.md` §8 suggests: at least six of the "open" known issues there are already fixed on HEAD (FTS sanitization, sections filter, label uniqueness, title rename, recordCommit aliases, irrelevant-restore). The biggest real gap is not a bug but a missing pillar: **automatic semantic links (related-on-read, tag TF-IDF, embeddings) do not exist at all**, and summary-first reading doesn't either — the "memories delivered automatically" half of the vision is 0% built. The biggest operational hazard is **stale `dist/` builds**: they caused all 3 test failures in this review *and* made `tim doctor` report a long-fixed broken orphan metric — the project repeatedly tests and runs old code while the source is already correct. Architecture-wise, 3 of the 10 packages are dead weight (tim-search is literally empty), and the HTTP/SSE daemon inherits per-process ambient state (cwd, global session cache) that makes multi-client use structurally unsafe.

## Improvements (prioritized)

### P0 — blocks trust or correctness

1. **Pretest build gate.** All spawn-based tim-mcp tests execute `dist/server.js`; a stale dist silently tests old code — this single mechanism produced every test failure and the wrong doctor output in this review. Add `npm run build` as a pretest step (or spawn from src via tsx) and rebuild/redeploy all dists now.
2. **Fix the summarizer hot-path trio:** wrap the finally-rollup in try/catch with `client.close()` in its own finally (summarize.ts:134-137); set lock TTL > timeout + margin; add a UNIQUE constraint per (summary-node, batch_index) so `writeBatchSummary` idempotency is structural, which also neutralizes the session-metadata RMW race's worst symptom.
3. **Decide the ambient-state model for the HTTP daemon before adding more MCP surface.** Session/project context must be per-connection (or explicit params required in HTTP mode); cwd- and global-cache-derived defaults are structurally wrong for multi-client. Same decision covers `project-schema.json` resolution.
4. **Give heuristic-fallback summaries a retry path.** Tag them (e.g. `#needs-resummarize`) and let the next summarizer run replace them; today every summarizer outage permanently pollutes long-term memory with raw dumps.
5. **Unify marker discovery to one policy** (vision says CWD-only) and make marker writes atomic (tmp+rename). Three components currently implement three rules; cross-project misbinding is the bug class TIM has been burned by repeatedly (P9999 incident).

## Architecture & Sensibility

**The 10-package split is about 60% justified.** tim-core (clean contract package), tim-store, tim-mcp, tim-cli, tim-hooks, tim-summarizer all earn their existence. Three do not: **tim-search contains zero source files** — it's a stub the architecture docs cite as "FTS5 + vector" while all search lives in tim-store; **tim-sync** is 226 lines whose only production-used export is `resolveLWW`, plus structurally broken Merkle dead code (no tree, no callers) whose passing tests lend false confidence; **tim-migrate** ships two overlapping engines, one of them untested and label-dropping. A package boundary that exists ahead of its second consumer is pure rent — delete or fold these until the feature actually arrives. tim-sync-client is real (crypto, queue, transport) and fine.

**SQLite + FTS5 is the right core.** WAL supports exactly the multi-process pattern TIM needs (session writer + background summarizer), the v6 expression indexes make JSON-metadata querying acceptable at this scale, and "theoretically infinite" via cold compression + sharding remains plausible on this base. Two structural caveats: store.ts is a 1758-line god class whose async facade over synchronous better-sqlite3 obscures transaction boundaries — that is precisely where the read-modify-write races come from (`runExclusive` exists; callers read state outside it); and `project-output.ts` is presentation living in the storage package. Frequently-queried JSON fields (`task.status`, `order`, `seq`, `kind`) will eventually deserve real columns.

**Hooks + external CLI summarizer: right idea, too many process boundaries.** Keeping extraction out of the agent's context window via detached background summarization is sound and local-first-appropriate, and the recent fixes went the right way — DB-derived counters and seq-range coverage as the source of truth instead of events. But every remaining failure class lives at a process boundary: shell string-splicing, coreutils `timeout` portability, SIGTERM-only reaping, a marker file with five unlocked writers. The summarizer lives in the same monorepo and could import tim-store directly — that one move deletes the MCP child process, the finally-leak, and the shell-quoting layer. Worth serious consideration before Phase 0.7 builds more on this pipeline.

**What to cut:** tim-search (now), tim-sync's dead half, legacy migrate.ts, tim_read_project, tim_lease (unless finished), the walk-up option. **What to double down on:** the plan-1..4 direction itself — single zod registry, internal-tool hiding, bind:false consolidation, deterministic LWW — this recent work is the best code in the repo; plus the 0.7 memory features (summary field, related-on-read), which are the difference between "a good project tracker" and the actual vision.

## Test & Ops Health

- **Suite:** 549/552 passed on the initial run; all 3 failures traced to a stale dist containing the removed PID lockfile (nondeterministic lock race between parallel spawn-based test files, both files pass in isolation). After rebuild, tim-mcp is green: 21 files / 158 passed / 2 skipped. Per-package runs during review: tim-store 251/251, tim-core 21/21, tim-migrate 21/21, tim-cli 31/31.
- **`tim doctor`** (read-only, against `~/.tim/tim.db`): 2916 entries, 10478 edges, FTS5 ✓ — but reported 4628 broken links and **7381 orphans (more than the total entry count)**, which is the signature of the *old* orphan metric; the fixed metric (dangling-parent check, store.ts:1643-1655) isn't in the installed build. Real orphan/broken-link numbers are therefore **unknown until rebuild** — rerun doctor after fixing the build gate, then decide on a cleanup tool.
- **Coverage blind spots:** all concurrency paths (lock-TTL steal, duplicate batch nodes, marker torn writes, logExchange vs writeBatchSummary, WAL contention); rollup failure modes; the 8 shell scripts in tim-hooks (zero tests — harness matrix, quoting); heuristic-fallback replaceability; sync-client conflict scenarios (exactly 1 integration test — no local-newer, no pagination, no mid-flush resume, no wrong-passphrase); suppression e2e across all read paths; legacy migrate.ts (zero tests); HTTP multi-client isolation; cli init/doctor/hook/checkpoint/sync subcommands.
- **Operational risks:** hmem-sync last synced 24d+ ago (per session banner); no DB backup primitive (known, INVENTORY-FIX-05) — that gap is now more urgent given the duplicate-label cleanup and orphan cleanup both imply write-heavy maintenance passes.
